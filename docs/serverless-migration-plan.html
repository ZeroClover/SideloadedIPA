01背景与目标

当前仓库的签名流水线（GitHub Actions + zsign）产出三类产物，全部依赖一台 Plesk 服务器（itms.zeroclover.io 所在 VPS）通过 scp/sshpass 发布：

  • 重签名后的 IPA 二进制（服务器本地路径，经 HTTPS 对外提供下载）；
  • 每个 app 的 itms.plist（itms-services 安装清单，与 IPA 同目录）；
  • 下载页面（site/ 静态文件：index.html + apps.js + app-card.js + app.js + styles.css，scp 到 docroot）。

本次迭代的目标是彻底移除对这台服务器的需求，转为无服务器架构：

  • 前端页面 + plist 托管到 Vercel：用 Next.js（App Router）重写下载页；plist 由 Vercel 上的 Route Handler 动态产出，URL 仍挂在自有域名下；
  • CI 按需更新：签名流水线每次构建后，自动更新前端数据源与 plist 内容 —— 通过「R2 上的 apps.json + Vercel 按需 ISR 重验证」实现，无需重新部署、无需回写提交；
  • IPA 上传到 Cloudflare R2：plist 内的下载链接指向 R2 的公开 URL（自定义域名），R2 出口流量免费；
  • 淘汰 ASSETS_SERVER_* 凭据与全部 scp 逻辑。
范围说明：本计划只改变「发布与托管」环节。签名本身（zsign、P12、mobileprovision 同步、缓存与变更检测逻辑）不受影响，保持原样。

02架构对比：现状 → 目标

现状 · 自建服务器

GitHub Actions (ubuntu) ├─ zsign 重签名 IPA ├─ scp ──► VPS (Plesk) │ ├─ /<dir>/App.ipa │ ├─ /<dir>/itms.plist │ └─ docroot: site/* 静态页 ├─ git commit 回写 site/apps.js └─ Cloudflare 代理缓存（需 ?v=N 手动破缓存） iOS Safari ─► itms.zeroclover.io（VPS） 页面 / plist / IPA 全部同源同机

目标 · 无服务器

GitHub Actions (ubuntu) ├─ zsign 重签名 IPA ├─ S3 API ──► Cloudflare R2 │ ├─ apps/<slug>/<ver>/App.ipa（版本化 key） │ ├─ apps/<slug>/icon.png │ └─ site/apps.json（数据源，合并更新） └─ HTTPS ──► Vercel /api/revalidate（带密钥） Vercel (Next.js 16, itms.zeroclover.io) ├─ / 下载页（RSC，读 apps.json，ISR） └─ /apps/<slug>/itms.plist Route Handler 动态生成 iOS Safari ─► Vercel 取页面 + plist └─► R2 自定义域名直取 IPA（不经 Vercel）

职责切分原则：Vercel 只承载 HTML 与几 KB 的 plist XML；IPA 字节流永远直连 R2（302/直链），不经过 Vercel 代理 —— 否则 Vercel 的 Fast Data Transfer 额度（Hobby ~100 GB/月）会被少数几次数百 MB 的下载瞬间耗尽。

03关键决策（含调研依据）

D1 · R2 上传使用 boto3（S3 兼容 API），而非 scp / wrangler / rclone

采用已确认在 pyproject.toml 增加 boto3 依赖（uv.lock 固定版本与哈希），endpoint 指向 https://<account_id>.r2.cloudflarestorage.com。签名 region 必须显式指定：R2 只接受自身的位置提示名（wnam/enam/weur/eeur/apac/oc）或 auto，boto3 默认拾取本机 AWS 配置（如 ap-northeast-1）会被 R2 以 InvalidRegionName 拒绝 —— 实现上以 auto 为默认、R2_REGION 环境变量固定为 bucket 的位置提示（本桶 apac）。

  • 与仓库「依赖全部锁定、可复现」的惯例一致（现有 zsign/asc 二进制均校验 SHA256，uv.lock 自带哈希）；
  • 纯 Python 流程，上传逻辑与现有 run_signing.py 同进程编排，pytest 可直接 mock client；
  • 备选：aws CLI（GitHub runner 镜像预装、零新依赖，但版本不受本仓库锁定）；rclone/wrangler 需要额外安装，对本场景无收益。

凭据：R2 API Token（Object Read & Write 权限，限定单个 bucket），GitHub Secrets 存 Access Key ID / Secret。

D2 · R2 公开访问用自定义域名，不用 r2.dev

采用已确认为 bucket 绑定自定义域 ipa.zeroclover.io（域名已在 Cloudflare，接入自动建 CNAME 并签发边缘证书）。

  • r2.dev 官方定位为开发用途：速率限制（数百 req/s 后 429）、可能限速、无边缘缓存，Cloudflare 明确建议生产用自定义域；
  • 对象 key 与 URL 路径一一对应；bucket 公开后按精确 key 提供下载，目录列举返回 404（不暴露清单）；
  • R2 出口流量免费（含自定义域与 S3 API），大 IPA 分发成本为零；注意免费/Pro/Business 套餐边缘缓存单文件上限 512 MB，超大 IPA 将始终回源 R2 —— 零出口费下可接受；
  • 该域名上不得开启 WAF 挑战/Bot Fight —— iOS 的 appstored 必须匿名访问（先发 HEAD 再 GET），任何登录墙/挑战页都会导致安装静默失败。

D3 · plist 由 Next.js Route Handler 动态生成，URL 留在 Vercel 域下

采用已确认web/app/apps/[slug]/itms.plist/route.ts —— App Router 支持带点号的静态段（官方文档示例即 rss.xml/route.ts），动态段 + itms.plist 组合得到 /apps/<slug>/itms.plist。

  • 满足「plist 本身托管到 Vercel」的要求，且数据只有一份（apps.json），不存在 CI 上传的 plist 与页面数据不一致的问题；
  • 响应头 Content-Type: text/xml; charset=utf-8（Apple 部署文档对 .plist 的规定类型；实测 iOS 对 MIME 宽容，但按官方来），Cache-Control: public, max-age=0, must-revalidate —— 清单必须永远新鲜，任何一层缓存旧版本都会导致装到旧版；
  • plist 内的 IPA URL 直接写 R2 自定义域地址（HTTPS、匿名可访问、支持 HEAD —— 均满足 Apple OTA 要求；清单与 IPA 不要求同宿主，App Center 时代即如此分发）；
  • 备测结论：itms-services 在 iOS 17/18 仍正常工作（Apple 官方文档 2025-09 仍在更新该分发方式；社区在 iOS 18.3 有实测记录），「iOS 17.2 移除 itms-services」系谣言。

D4 · 「按需更新」用 R2 apps.json + ISR 按需重验证，不用 Deploy Hook / 提交回写

采用已确认CI 构建完成 → 上传 IPA + 合并更新 R2 上的 site/apps.json → curl 调用 Vercel 上的 /api/revalidate（共享密钥）→ 路由内执行 revalidateTag('apps', 'max')。apps.json 只存在于 R2，不进 git、不做提交回写。

  • 页面与 plist 路由统一用带 { next: { tags: ['apps'] } } 的 fetch 读 apps.json；重验证后下一次请求即取新数据，更新延迟 ≈ 一次请求，零构建分钟消耗；
  • 这是 Next.js 官方针对「数据在部署之外变化」的推荐模式；revalidatePath/revalidateTag 对 Route Handler 同样有效（官方 API 参考明确说明）；
  • 落选：Deploy Hook触发的是整分支重建（apps.json 在 R2 上，重建也拿不到新数据），徒耗构建；
  • 落选：提交回写触发 Git 自动部署每次签名产生一次完整 Vercel 构建 + 提交噪音（现状 apps.js 回写即此模式，正好借机去掉；contents: write 权限可收回为 read）。

D5 · IPA 绝不经过 Vercel 代理

采用plist 内的 IPA URL、页面上的任何直链，全部指向 R2 自定义域。禁止用 Vercel rewrites 代理 R2 的 IPA —— rewrite 到外部地址是代理（Vercel 拉源站字节再转发），全部计入 Fast Data Transfer。若未来需要 Vercel 域下的 IPA 短链，只能用 302 重定向。

D6 · DNS：Vercel 域名在 Cloudflare 侧必须灰云（DNS only）

采用itms.zeroclover.io → CNAME cname.vercel-dns.com，关闭橙色云代理。橙云开启会导致 Vercel 域名校验看到 Cloudflare IP 报 "Invalid Configuration"，且证书签发失败（525 握手错误）。R2 自定义域不受影响（本来就是 Cloudflare 自家产品）。

D7 · IPA 使用版本化 key + immutable 缓存，旧版本由 CI 清理

采用已确认apps/<slug>/<version>/<App>.ipa，上传时 Content-Type: application/octet-stream（Apple 对 .ipa 的规定类型）、Content-Disposition: attachment（浏览器直开时存为文件而非跳转）、Cache-Control: public, max-age=31536000, immutable。

  • 版本化 key = 原子切换（apps.json 一翻指向即全量生效）、永不需要 purge；同时避开 R2「同 key 每秒 1 次写」的限制；
  • 不配置 bucket 生命周期规则：很多 App 不保证固定发版周期，按时间过期（如 90 天）会误删仍被引用的唯一版本。改为 CI 在每次构建末尾清理：新 IPA 上传 → apps.json 切换指向 → revalidate 成功之后，删除该 slug 下不再被 apps.json 引用的旧版本 key。删除白名单以 apps.json 的引用集合为准（与时间无关），只处理本次构建涉及的 slug；任一步失败即跳过清理。
  • 免费额度 10 GB 存储 + 每月 1M A 类 / 10M B 类操作；清理使存储占用恒定（每 app 一份当前版），本场景绰绰有余。

04目标架构详细设计

4.1 R2 数据布局与 apps.json 数据契约

Bucket 布局（单 bucket，如 zeroclover-ipa）：

apps/<slug>/icon.png                # 卡片图标（一次性从服务器迁移；后续可由 CI 从 IPA 提取）
site/apps.json                       # 页面 + plist 的唯一数据源

apps.json（由 CI 全权生成与合并更新；已决议：不保留任何手工维护条目，仅收录流水线 task 产出）：

slug 取 task 的 slug 字段（缺省 slugify(app_name)，与旧 apps.js 的 dir 对齐，如 JHenTai、fehviewer），流水线按 slug 写入/刷新条目；name 直接取 tasks.toml 的 app_name，不再有「策展名保留」逻辑。apps.json 只存在于 R2，是页面与 plist 的唯一数据源，不进 git、不做提交回写。

4.2 Next.js 前端（web/）

Next.js 16（当前稳定版）+ React 19 + TypeScript，Node 运行时（ISR/按需重验证仅 Node 运行时支持；不开启 cacheComponents）。视觉 1:1 移植现有设计令牌（暖白 #f4f4f2、近黑 #1a1a18、Space Grotesk + Space Mono，用 next/font/google 自托管，去掉运行时对 Google Fonts CDN 的依赖）。

web/
├── app/
│   ├── layout.tsx                       # 字体 + 全局样式
│   ├── page.tsx                         # RSC：读 apps.json → <AppGrid>（客户端组件，负责搜索过滤）
│   ├── globals.css                      # 移植 styles.css 设计令牌
│   ├── apps/[slug]/itms.plist/route.ts  # GET → plist XML（text/xml, no-cache）
│   └── api/revalidate/route.ts          # CI Webhook：验密钥 → revalidateTag('apps','max')
├── lib/
│   ├── apps.ts                          # getApps(): fetch apps.json, { next: { tags: ['apps'] } }
│   └── plist.ts                         # buildItmsPlist()：移植 Python 版 XML 格式（4 空格缩进）
├── components/AppCard.tsx               # 移植 app-card.js（去 Shadow DOM，样式进 globals.css）
├── public/
├── next.config.ts

plist 路由要点（Next 16 中 params 为 Promise，必须 await）：

// app/apps/[slug]/itms.plist/route.ts
export async function GET(_req: Request, { params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const app = (await getApps()).find(a => a.slug === slug);
  if (!app) return new Response('not found', { status: 404 });
  return new Response(buildItmsPlist(app), {
    headers: {
      'Content-Type': 'text/xml; charset=utf-8',
      'Cache-Control': 'public, max-age=0, must-revalidate',
    },
  });
}

安装按钮：itms-services://?action=download-manifest&url= + encodeURIComponent(plistUrl)（URL 含特殊字符时必须百分号编码）。仅 iOS Safari 生效（桌面点击无反应为预期）。

revalidate 路由：校验 ?secret= 与环境变量 REVALIDATE_SECRET 一致后 revalidateTag('apps', 'max')（Next 16 需第二参数 cacheLife），返回 { revalidated: true }。

图标：<img> 直引 R2 的 iconUrl，保留现有「加载失败退回首字母色块」的兜底。

06风险与缓解

风险缓解
Cloudflare 橙云代理导致 Vercel 域校验/证书失败CNAME 必须灰云（DNS only），切流检查清单中显式列出（D6）
IPA 误走 Vercel 代理，耗尽 Fast Data Transferplist/页面只放 R2 直链；代码评审检查项（D5）；Vercel 用量告警
plist 被某层缓存导致装到旧版plist 响应 max-age=0, must-revalidate；Vercel 边缘对未哈希静态/动态响应默认短缓存；真机验证
R2 自定义域上的 WAF/Bot 拦截 appstored该域关闭挑战类功能；P4 真机验证即覆盖
apps.json 更新与 IPA 上传之间的短暂不一致顺序：先传 IPA（immutable 新 key）→ 再传 apps.json → 最后 revalidate；最差情况是页面暂时指向尚未刷新的旧版本，不会指向不存在的文件
revalidate 密钥泄漏 → 任意人触发缓存重建仅造成额外 fetch，无数据风险；密钥入 GitHub/Vercel 双端 secrets，泄漏即轮换
>512 MB IPA 不被边缘缓存R2 零出口费，回源直发可接受；无需处理
旧版本 IPA 无限积累CI 清理：每次构建末尾删除不被 apps.json 引用的旧版本 key（D7）
清理误删仍被引用的版本（安装中 404）严格顺序「上传新 IPA → 更新 apps.json → revalidate → 清理」；白名单以 apps.json 引用集合为准而非时间；任一步失败即跳过清理
iOS 18 侧载固有限制（信任描述文件、可能需重启）与现状相同的平台行为，非本次迁移引入；页面 FAQ 说明
Vercel Hobby 套餐限制（非商业、部署/函数配额）KB 级响应 + 低频构建远低于配额；使用现有账号导入项目（已决议，§9）

08回滚方案

  • P5 切流前：任何问题 → 不切换 DNS，现状完全不受影响（R2/Vercel 均为新增设施）。
  • P5 切流后 48h 内：DNS 改回 VPS 原记录即可（服务器在 P6 前保持原样只读运行）；TTL 建议切流前调低至 300s。
  • P6 后：服务器已退役则无法整体回滚；故 P6 以「连续 2 周无异常」为前提执行。代码层面回滚 = revert 合并提交。

09开放问题决议（2026-07-18 全部确认）

  1. Vercel 套餐与账号：决议 —— 使用现有 Vercel 账号导入项目，无其他待确认项。
  2. site/ 旧目录：决议 —— 不再保留、不归档；P6 随服务器退役一并删除。
  3. 手工 app：决议 —— 不维护任何先前的手动维护 app。对照 tasks.toml（5 个 task）与现状 apps.js（7 个条目），EhPanda、FEhViewer、Sonolus 三个手工条目在迁移后下线；页面仅展示流水线 task 产出，apps.json 由 CI 全权生成，无任何手工条目保留/编辑机制。
  4. 图标策略：决议 —— P1 一次性迁移现有图标（仅 5 个流水线 app）；不实现 CI 从 IPA 自动提取图标。
